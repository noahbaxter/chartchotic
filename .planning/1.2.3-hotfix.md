### 5. Fix native framerate cap on multi-monitor

Monty reports: Native framerate setting maxes at 60Hz instead of 144Hz on his Nvidia multi-monitor setup (main monitor 144, secondary 60). 0.9.4 worked fine at 144. Only report so far, but it's a regression from the pre-native-refresh era.

- Fix: use `juce::Desktop::getDisplays().getDisplayForRect(editorBounds)` to find the display the plugin editor is actually rendering into, then read that display's refresh rate.

### 6. Fix REAPER hang on track duplication ⚠️ CRITICAL ✅ DONE

Invontor: duplicating a REAPER track with the Chartchotic plugin attached hangs REAPER. Eventually starts playing smoothly while still hung; required force-quit.

**Root cause:** `ReaperMidiPipeline::process()` was called from the audio thread and invoked REAPER API (`MIDI_GetTrackHash`, full media-item iteration). REAPER's C API is main-thread-only for project-state functions. Single-instance masked the violation (no contention); track duplication exposed it — REAPER holds its track-list lock during the copy, the audio thread blocks in a REAPER call, and the main thread (editor VBlank polling) jams on the same locks. Audio eventually recovers; UI stays hung.

**Fix:** Branch `fix/reaper-duplicate-hang`, commit `da1d115`. Audio thread now only touches cached data. New `ReaperMidiPipeline::pollForHashChange()` runs on the editor's 50ms VBlank tick (any transport state, hash-gated). `JUCE_ASSERT_MESSAGE_THREAD` guards at the four heavy REAPER API entry points catch future regressions in debug builds.

**Verified:** 6× rapid duplicate in REAPER with Chartchotic attached — no hang. Performance suffers at N instances on one project (shared-cache opportunity deferred; pop-out highway windows added to BACKLOG as the real fix).

## Blocked on decision

### RenderStyle refactor (tech debt, surfaced during task 1 work)

While working on task 1, noticed `drawGem` has 5+ redundant `isDrumLike(activePart)` / `bool isDrums` declarations (lines 114, 116, 128, 190, 198, 285, 287, 302 in `NoteRenderer.cpp` — some shadow each other). Root cause: the binary `isDrums` pattern is overloaded across data identity, behavior branches, and lane count.

**History:**
- `1f08c0a` — `RenderType` enum added (`FIVE_FRET`, `FOUR_LANE_DRUMS`, `FIVE_LANE_DRUMS`, `ELITE_DRUMS`, etc.) with `getRenderType(Part)` mapping.
- `5e177cf` — "Mechanical replacement" commit: `activePart == Part::DRUMS` → `isDrumLike(activePart)`. Explicitly scoped to "no behavioral changes." Did not touch paired data tables or the binary collapse pattern.
- Net result: `RenderType` enum exists but renderers still branch on `isDrumLike()` which collapses it back to a bool.

**Proposed refactor (phases 1–3 low-risk, phase 4 invasive):**
1. Define `RenderTypeConfig` struct bundling per-style data (lane count, coord tables, offsets, perspective params, Bemani config, asset selector). `getRenderTypeConfig(RenderType)` returns configs for the two implemented styles.
2. Migrate renderer data selection from `isDrums ? drumX : guitarX` to `config.X`, one file per commit. Keep `isDrums` for behavior branches (cymbals, HOPOs).
3. `isDrums` cleanup: hoist single declaration to top of each function, delete duplicates/shadows.
4. (Deferred) Add behavior flags (`supportsCymbals`, `supportsHopos`) to config, replace behavior branches. This is what unlocks keyboard/5-lane drums cleanly.

**Scope:** ~35 files, ~62 `isDrumLike`/`isGuitarLike` call sites, 15+ paired data tables across `PositionConstants.h`, `BemaniConfig.h`, renderers, `AssetManager`, `TrackResolver`.

**Open decision — version strategy:**
- (a) Ship 1.2.3 with just the bug fixes from tasks 2–6. Land refactor on a separate branch into 1.3.0.
- (b) Do phases 1–3 before 1.2.3 ships and bundle it all as 1.3.0 (bug fixes become minor changelog items).
- (c) Do phases 1–3 on a branch, merge after 1.2.3 ships.

Noah's lean as of last session: asked to do the refactor "first before moving onto this next rendering thing" — suggests option (b) or (c). Version label TBD.

**`wip/authoring` branch state (checked 2026-04-11):**
Contains `5879f6f` "painter refactor: separate drawing from orchestration" and `f74e36a` "NoteRenderContext struct". These extracted painter classes (`NotePainter`, `LanePainter`, etc.) from renderers and introduced `NotePainter::NoteRenderContext` which carries per-frame render state including `isDrums`, populated once per frame.

**Net effect of wip/authoring on the isDrums mess:** `NoteRenderer.cpp` dropped from ~7 `isDrumLike`/`isDrums` sites (on dev) to 2 (on wip/authoring) — both just populating the context field. That's a partial Phase 3 win.

**What wip/authoring did NOT touch:** `PositionConstants.h` paired tables are unchanged. `RenderType` enum still collapses to bool via `isDrumLike`/`isGuitarLike`. Phases 1, 2, and the rest of Phase 3 are all still ahead.

**Sequencing recommendation:** wip/authoring is mid-flight (tip commit today). Merging it FIRST and then doing the RenderStyle refactor on top is probably cleaner than the reverse — `NoteRenderContext` is a natural home for a `const RenderTypeConfig* config` field, and rebasing the authoring work onto a bigger structural refactor would fight active development.
